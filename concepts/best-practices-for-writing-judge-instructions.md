# Best Practices for Writing Judge Instructions

**Judge instructions** are the natural language prompts that define how an LLM-based judge evaluates the quality of GenAI agent outputs. Well-crafted instructions are essential for obtaining reliable, consistent, and actionable evaluation results.

## Overview

Custom LLM judges allow you to define complex and nuanced scoring guidelines for GenAI applications using natural language. While MLflow provides built-in LLM judges for common quality dimensions, custom judges created using `make_judge()` give you full control over evaluation criteria. The quality of your evaluation depends heavily on how well you write your judge instructions. ^[custom-judges-databricks-on-aws.md]

## Be Specific About Expected Output Format

Your instructions must clearly specify what format the judge should return. The format choice depends on the type of evaluation you need: ^[custom-judges-databricks-on-aws.md]

- **Categorical responses**: List specific values the judge should select from (for example, `'fully_resolved'`, `'partially_resolved'`, `'needs_follow_up'`)
- **Boolean responses**: Explicitly state the judge should return `true` or `false`
- **Numeric scores**: Specify the scoring range and what each score means

Clear output formatting ensures that evaluation results are consistent and machine-readable across different runs and agent configurations. ^[custom-judges-databricks-on-aws.md]

## Break Down Complex Evaluations

For complex evaluation tasks, structure your instructions into clear sections. This helps the LLM judge understand the evaluation framework and apply it consistently: ^[custom-judges-databricks-on-aws.md]

1. **What to evaluate** – Define the specific quality dimension or behavior being assessed
2. **What information to examine** – Specify which template variables (inputs, outputs, expectations, or trace) the judge should analyze
3. **How to make the judgment** – Provide step-by-step reasoning guidelines or criteria
4. **What format to return** – Specify the output format (categorical, boolean, or numeric) and the exact values or range

Structuring instructions this way reduces ambiguity and improves the consistency of judge ratings across different evaluation runs. ^[custom-judges-databricks-on-aws.md]

## Understand Template Variables

Your instructions must include at least one template variable to access agent data, but you don't need to use all of them. The following template variables are available: ^[custom-judges-databricks-on-aws.md]

- `{{ inputs }}` – Input data provided to the agent
- `{{ outputs }}` – Output data generated by your agent
- `{{ expectations }}` – Ground truths or expected outcomes
- `{{ trace }}` – The complete execution trace of your agent (required for trace-based judges)

Only these four template variables are allowed. Custom variables like `{{ question }}` will throw validation errors in order to ensure consistent behavior and prevent template injection issues. ^[custom-judges-databricks-on-aws.md]

## Model Requirements for Trace-Based Judges

When writing instructions for [Trace-based Judges](/concepts/trace-based-judges.md), you must consider that these judges require a capable model for trace analysis. If your instructions use the `{{ trace }}` template variable, you must specify the `model` argument in `make_judge()`. Recommended models include: ^[custom-judges-databricks-on-aws.md]

- `databricks:/databricks-gpt-5-mini`
- `databricks:/databricks-gpt-5`
- `databricks:/databricks-gpt-oss-120b`
- `databricks:/databricks-claude-opus-4-5`

For trace-based judges, include instructions for analyzing execution traces, validating tool usage patterns, identifying performance bottlenecks, investigating execution failures, and verifying multi-step workflows. ^[custom-judges-databricks-on-aws.md]

## Align Judges with Human Experts

The base judge instructions are a starting point. As you gather expert feedback on your application's outputs, you can align the LLM judges to the feedback to further improve judge accuracy. This iterative process helps ensure that the judge's assessments match human quality judgments. See Align judges with human feedback for details. ^[custom-judges-databricks-on-aws.md]

## Example of Well-Structured Instructions

The following example demonstrates best practices by defining a judge with clear evaluation criteria and output format for assessing tool calling correctness: ^[custom-judges-databricks-on-aws.md]

```python
from mlflow.genai.judges import make_judge
from typing import Literal

tool_usage_judge = make_judge(
    name="tool_usage_validator",
    instructions=(
        "Analyze the {{ trace }} to verify correct tool usage.\n\n"
        "Check that the agent selected appropriate tools for the user's request "
        "and called them with correct parameters."
    ),
    feedback_value_type=Literal["correct", "incorrect"],
    model="databricks:/databricks-gpt-5-mini"  # Required for trace-based judges
)
```

This example specifies what to evaluate (tool usage), what information to examine (the trace), how to make the judgment (check tool selection and parameter correctness), and the exact output format (categorical values "correct" or "incorrect"). ^[custom-judges-databricks-on-aws.md]
