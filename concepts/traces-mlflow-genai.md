# Traces (MLflow GenAI)

**Traces** are the core observability primitive in MLflow for GenAI. They capture the complete execution of a generative AI application, including inputs, outputs, and every intermediate step such as LLM calls, retrievals, and tool use. ^[concepts-data-model-databricks-on-aws.md]

## Overview

Traces are created automatically for every execution of an application, both in development and production. Each trace can be linked to the specific application version ([logged model](/concepts/loggedmodel.md)) that generated it. Traces also carry [Assessments](/concepts/assessments.md) — quality measurements that include feedback from automated [[scorers]], end users, and domain experts, as well as ground truth expectations provided by domain experts. ^[concepts-data-model-databricks-on-aws.md]

Traces are the foundational data that enables both observability and systematic quality evaluation. They are used to observe and debug application behavior (such as latency and cost) and to create [Evaluation Datasets](/concepts/evaluation-datasets.md) from production logs that can be reused in offline quality assessments. ^[concepts-data-model-databricks-on-aws.md]

## Anatomy of a Trace

A trace records:

- **Inputs** – The user request or prompt passed to the application.
- **Outputs** – The final response generated by the application.
- **Intermediate steps** – Every invocation of internal components, such as LLM calls, context retrieval, function calls, and tool invocations, along with their results.

This level of detail lets developers inspect the exact sequence of operations that led to a given output. ^[concepts-data-model-databricks-on-aws.md]

## Creating and Logging Traces

MLflow provides SDKs and automatic instrumentation to log traces. In most cases no manual logging code is needed; MLflow’s tracing libraries automatically capture execution data when an instrumented app runs. Developers can also manually log traces using the `mlflow.tracing` API. See the Tracing quickstart and [Instrument your app](/concepts/mlflow-instrumentation-guidance.md) guides for implementation details. ^[concepts-data-model-databricks-on-aws.md]

## Using Traces for Evaluation

### Scorers and Assessments

[[Scorers]] are functions that analyze a trace and produce feedback assessments. A scorer extracts relevant data from the trace — such as the request, response, and retrieved context — and evaluates it using either deterministic logic or an [LLM judge](/concepts/llm-judges.md). The same scorer can be used both during development evaluation and production monitoring. ^[concepts-data-model-databricks-on-aws.md]

### Trace-Based Judges

When using custom judges via the `make_judge()` API, the full execution trace can be passed into the judge’s instructions by including the `{{ trace }}` placeholder. This allows judges to analyze tool invocations, intermediate reasoning steps, and their results, not just the final input and output. Trace-based judges require a model specification (e.g., `model="databricks:/databricks-gpt-5-mini"`). ^[create-a-custom-judge-using-make_judge-databricks-on-aws.md]

### Evaluation in Development

The `mlflow.genai.evaluate()` function runs an application version against an [Evaluation Dataset](/concepts/evaluation-dataset.md) and a set of scorers. It produces an evaluation run containing the traces generated by the app and the feedback assessments attached to each trace. The results help developers validate improvements and identify regressions. ^[concepts-data-model-databricks-on-aws.md]

### Evaluation in Production

Production monitoring uses scheduled scorers (via `mlflow.genai.Scorer.start()`) to automatically evaluate traces from a deployed application. The scorers run on production traces, attach feedback assessments, and allow teams to detect quality issues in real time. ^[concepts-data-model-databricks-on-aws.md]

## Traces in the Data Model

Traces live inside an [MLflow Experiment](/concepts/mlflow-experiment.md), which serves as a container for all data related to a single GenAI application. The experiment holds not only traces but also evaluation runs, logged models, prompts, labeling sessions, and evaluation datasets. ^[concepts-data-model-databricks-on-aws.md]

The following diagram summarizes the data model relationships (see the full data model section in the source for details):

```
Experiment
├── Traces (app execution logs)
│   └── Assessments (quality measurements)
├── Evaluation data
│   ├── Evaluation datasets
│   └── Evaluation runs
├── Human labeling data
│   ├── Labeling sessions
│   └── Labeling schemas
└── Application versioning data
    ├── Logged models
    └── Prompts
```

Traces are the only required entity; all other parts of the data model are optional. ^[concepts-data-model-databricks-on-aws.md]

## User Interfaces

The [MLflow Experiment UI](/concepts/mlflow-experiment.md) allows users to search for and view traces, review attached feedback and expectations, and analyze evaluation results. The Review app is a separate web UI where domain experts label traces from labeling sessions. ^[concepts-data-model-databricks-on-aws.md]
