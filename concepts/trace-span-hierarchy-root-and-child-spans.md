# Trace Span Hierarchy (Root and Child Spans)

**Trace span hierarchy** is the structural organization of spans within a [trace (observability)|trace](/concepts/mlflow-tracing-for-genai-observability.md), where a single root span represents the top-level operation and child spans represent nested sub-operations. This hierarchy reveals the parent–child relationships between components of an application, making it possible to understand how individual steps contribute to the overall request flow.

## Root Span

The root span is the first span created in a trace. It represents the entry point of the application and typically captures the inputs to the top-level function. In the example from the [MLflow Tracing](/concepts/mlflow-tracing.md) quickstart, the root span corresponds to the `my_app(input)` function call. The root span's attributes include the original input (the user's question) and the final output (the model's response). ^[get-started-mlflow-tracing-for-genai-databricks-notebook-databricks-on-aws.md]

## Child Spans

Child spans are nested inside the root span and represent sub-operations that are executed as part of the top-level request. Each child span is linked to its parent span by a parent–child relationship. In the quickstart example, there is a single child span that represents the OpenAI completion request (`client.chat.completions.create`). This child span captures details such as the model name, number of tokens used, and timing information. ^[get-started-mlflow-tracing-for-genai-databricks-notebook-databricks-on-aws.md]

## Attributes Captured in the Hierarchy

Both root and child spans carry metadata called **attributes**. The trace shown in the quickstart includes:

- **Model name** (e.g., `databricks-claude-sonnet-4`)
- **Token counts** (affecting cost)
- **Timing information** (how long each operation took)
- **Inputs** (the messages sent to the model)
- **Outputs** (the response received from the model)

These attributes are organized by span: the root span holds the application-level inputs and outputs, while the child span holds the model‑call‑specific details. ^[get-started-mlflow-tracing-for-genai-databricks-notebook-databricks-on-aws.md]

## Example Trace from [MLflow Tracing](/concepts/mlflow-tracing.md)

The following structure is observed in the trace generated by the quickstart notebook:

```
Root span (my_app)
  └─ Child span (OpenAI completion request)
```

The root span shows the function input (`"What is MLflow?"`) and the final printed output. The child span shows the model name, token usage, and duration of the API call. This hierarchy provides a clear picture of the application's behavior: what was asked, what was returned, how long the request took, and how many tokens were consumed. ^[get-started-mlflow-tracing-for-genai-databricks-notebook-databricks-on-aws.md]

## Value for Complex Applications

For simple applications, the root‑child hierarchy is straightforward. For more complex applications—such as [Retrieval Augmented Generation (RAG)](/concepts/retrieval-augmented-generation-rag.md) systems or multi‑step agents—the trace span hierarchy becomes more valuable because it reveals the inner workings of each component and step. Multiple child spans can be nested, each representing a different sub‑operation (e.g., a retrieval call, an LLM call, a tool invocation). ^[get-started-mlflow-tracing-for-genai-databricks-notebook-databricks-on-aws.md]
