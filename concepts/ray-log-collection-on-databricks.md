# Ray Log Collection on Databricks

**Ray Log Collection on Databricks** refers to the mechanism for capturing logs generated by a [Ray cluster](/concepts/global-mode-ray-cluster.md) running on a Databricks Spark cluster, ensuring that debugging and monitoring information is preserved after the cluster shuts down.

## Setting a Log Output Location

You can configure the destination for Ray cluster logs by passing the `collect_log_to_path` argument to the `setup_ray_cluster` API (or `setup_global_ray_cluster`). The value is a path string specifying where log files should be written. ^[create-and-connect-to-ray-clusters-on-databricks-databricks-on-aws.md]

```python
from ray.util.spark import setup_ray_cluster

setup_ray_cluster(
    max_worker_nodes=2,
    collect_log_to_path="/dbfs/path/to/ray_collected_logs"
)
```

## Log Collection Behavior

Log collection runs automatically after the [Ray cluster](/concepts/global-mode-ray-cluster.md) is shut down. If no `collect_log_to_path` is set, logs are stored only in the local storage of the cluster’s worker and head nodes. This local storage is deleted when the Spark cluster is terminated, making logs unrecoverable. ^[create-and-connect-to-ray-clusters-on-databricks-databricks-on-aws.md]

## Recommended Paths

Databricks recommends setting a path that starts with `/dbfs/` (pointing to the [DBFS root](/concepts/dbfs-root-location.md)) or a Unity Catalog Volume path. Using a durable storage location preserves logs even if the Spark cluster is terminated, enabling post-mortem analysis and long-term retention. ^[create-and-connect-to-ray-clusters-on-databricks-databricks-on-aws.md]

## Troubleshooting Notes

Log collection is independent of the Ray memory monitor issues that can cause OOM-related task failures. If you inadvertently disable the memory monitor (by setting `RAY_memory_monitor_refresh_ms=0`), log output location configuration remains unaffected. ^[create-and-connect-to-ray-clusters-on-databricks-databricks-on-aws.md]
