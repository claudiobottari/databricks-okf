# Custom Code-Based Scorers (MLflow GenAI)

**Custom Code-Based Scorers** are user-defined evaluation functions in [MLflow GenAI](/concepts/mlflow-3-for-genai.md) that compute metrics for AI agent or application outputs. Unlike [LLM-as-a-Judge](/concepts/llm-as-a-judge.md) scorers that rely on a separate language model for evaluation, code-based scorers implement arbitrary Python logic to assess outputs, giving developers full control over the evaluation criteria. ^[code-based-scorer-examples-databricks-on-aws.md]

## Overview

Code-based scorers allow you to define flexible evaluation metrics for your AI agent or application. They can access various data sources including the full [[MLflow Trace]] object, inputs, outputs, and expectations (ground truth values). Scorers can return primitive values (bool, int, float, str) or structured [Feedback](/concepts/feedback-object.md) objects with rationale and error information. ^[code-based-scorer-examples-databricks-on-aws.md]

## Defining a Scorer

### Using the `@scorer` Decorator

The primary way to define a code-based scorer is with the `@scorer` decorator from `mlflow.genai.scorers`. The decorated function receives parameters based on the data it needs to access. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
from mlflow.genai.scorers import scorer
from mlflow.entities import Feedback

@scorer
def my_custom_scorer(outputs: str) -> Feedback:
    # Evaluation logic here
    return Feedback(value=True, rationale="Good quality")
```

### Using the `Scorer` Base Class

For scorers that require state or more complex configuration, use the [Scorer base class](/concepts/scorer-class.md). This is a Pydantic object that supports additional fields and is useful for offline evaluation. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
from mlflow.genai.scorers import Scorer
from mlflow.entities import Feedback

class ResponseQualityScorer(Scorer):
    name: str = "response_quality"
    min_length: int = 50

    def __call__(self, outputs: str) -> Feedback:
        # Evaluation logic here
        return Feedback(value=True, rationale="Meets criteria")
```

**Note:** Class-based `Scorer` subclasses are supported for offline evaluation with `mlflow.genai.evaluate()` only. They cannot be registered for [Production Monitoring](/concepts/production-monitoring.md). To use custom scorers in production monitoring, use the `@scorer` decorator. ^[code-based-scorer-examples-databricks-on-aws.md]

## Scorer Parameters

Scorers can accept the following parameters, which MLflow automatically injects based on the function signature: ^[code-based-scorer-examples-databricks-on-aws.md]

| Parameter | Description |
|-----------|-------------|
| `inputs` | The input data passed to the AI application (e.g., dictionary with `messages`) |
| `outputs` | The output generated by the AI application |
| `trace` | The full [[MLflow Trace]] object for accessing spans, timing, and execution details |
| `expectations` | Ground truth values or labels provided in the evaluation dataset |

## Return Types

### Primitive Values

Scorers can return simple Python types like `bool`, `int`, `float`, or `str`. The metric name defaults to the scorer function name. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
@scorer
def exact_match(outputs: str, expectations: dict) -> bool:
    return outputs == expectations["expected_response"]
```

### Single Feedback Object

Return a [Feedback](/concepts/feedback-object.md) object for structured evaluation with rationale and optional error information. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
@scorer
def quality_check(outputs: str) -> Feedback:
    return Feedback(
        value=True,
        rationale="Response meets quality criteria"
    )
```

### Multiple Feedback Objects

A single scorer can return a list of `Feedback` objects, allowing one scorer to assess multiple quality facets simultaneously. Each `Feedback` object must have a unique `name`, which becomes the metric name in the results. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
@scorer
def comprehensive_check(outputs: str) -> list[Feedback]:
    return [
        Feedback(name="is_not_empty", value="yes" if outputs else "no"),
        Feedback(name="char_length", value=len(outputs)),
    ]
```

## Common Patterns

### Accessing Trace Data

Scorers can access the full trace object to examine spans, timing, and execution details. This is useful for performance-related metrics. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
from mlflow.entities import Trace, SpanType

@scorer
def response_time_check(trace: Trace) -> Feedback:
    llm_span = trace.search_spans(span_type=SpanType.CHAT_MODEL)[0]
    response_time = (llm_span.end_time_ns - llm_span.start_time_ns) / 1e9
    # Evaluation logic
```

### Wrapping Built-in Judges

Custom scorers can wrap [Built-in LLM Judges](/concepts/built-in-llm-judges.md) to preprocess data or post-process feedback. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
from mlflow.genai.judges import is_context_relevant

@scorer
def custom_relevance_check(inputs: dict, outputs: str) -> Feedback:
    last_user_message = extract_last_user_message(inputs)
    return is_context_relevant(
        request=last_user_message,
        context={"response": outputs},
    )
```

### Using Expectations

Expectations provide ground truth values for offline evaluation. Scorers can access them to compute accuracy metrics. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
@scorer
def keyword_presence(outputs: str, expectations: dict) -> Feedback:
    expected_keywords = expectations.get("expected_keywords", [])
    missing = [k for k in expected_keywords if k.lower() not in outputs.lower()]
    # Return feedback based on keyword presence
```

### Integrating Custom LLM Judges

Scorers can make their own LLM API calls for evaluation, giving full control over the judging process. Set the `source` field in the `Feedback` object to indicate the assessment source. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
@scorer
def answer_quality(inputs: dict, outputs: str) -> Feedback:
    # Call your own LLM for evaluation
    judge_response = client.chat.completions.create(...)
    return Feedback(
        value=parsed_score,
        rationale=parsed_rationale,
        source=AssessmentSource(
            source_type=AssessmentSourceType.LLM_JUDGE,
            source_id="claude-sonnet-4-5",
        )
    )
```

### Conditional Logic with Guidelines

Scorers can apply different [Guidelines judges](/concepts/guidelines-llm-judges.md) based on user attributes or other context. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
from mlflow.genai.scorers import Guidelines

@scorer
def tier_based_validator(inputs, outputs):
    user_tier = inputs.get("user_tier", "standard")
    if user_tier == "premium":
        return premium_judge(inputs=inputs, outputs=outputs)
    else:
        return standard_judge(inputs=inputs, outputs=outputs)
```

## Error Handling

MLflow surfaces scorer errors gracefully. Scorers can use two approaches: ^[code-based-scorer-examples-databricks-on-aws.md]

1. **Return an error Feedback object**: Include an `AssessmentError` in the `Feedback` object.
2. **Raise an exception**: Let MLflow handle the error and continue evaluation.

```python
from mlflow.entities import AssessmentError

@scorer
def resilient_scorer(outputs):
    try:
        # Evaluation logic
        return Feedback(value=True, rationale="Valid")
    except Exception as e:
        # Option 1: Return error feedback
        return Feedback(
            value=None,
            error=AssessmentError(
                error_code="PROCESSING_ERROR",
                error_message=str(e)
            )
        )
        # Option 2: Raise exception
        # raise
```

## Naming Conventions

Metric names follow these rules: ^[code-based-scorer-examples-databricks-on-aws.md]

- **Primitive value or unnamed Feedback**: The scorer function name becomes the metric name.
- **Single named Feedback**: The name specified in the `Feedback` object is used.
- **Multiple Feedback objects**: Names specified in each `Feedback` object are preserved; each must be unique.

## Chaining Evaluation Results

Scorers can be used in iterative evaluation workflows. After identifying problematic traces with one scorer, you can collect those traces and re-evaluate with different scorers or an updated application. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
# Initial evaluation
results1 = mlflow.genai.evaluate(data=traces, scorers=[Safety()])

# Filter problematic traces
safety_failures = mlflow.search_traces(run_id=results1.run_id)
# Filter and re-evaluate
results2 = mlflow.genai.evaluate(data=safety_failures, scorers=[Guidelines(...)])
```
