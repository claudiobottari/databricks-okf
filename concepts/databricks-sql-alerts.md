# Databricks SQL Alerts

**Databricks SQL Alerts** is a notification framework within Databricks that monitors query results and triggers actions when specified conditions are met. Alerts are built on top of Databricks SQL queries and can be configured to run on a schedule, evaluate a condition, and send notifications to email, webhooks, Slack, PagerDuty, or other applications. ^[profile-alerts-databricks-on-aws.md]

## Common Use Cases

While alerts can be used with any Databricks SQL query, a common application is monitoring data quality through profile metrics. Examples include:

- Receiving a notification when the fraction of missing values in a column exceeds a threshold.
- Detecting changes in data distribution using the [Drift Metrics Table](/concepts/drift-metrics-table.md).
- Identifying data drift relative to a [Baseline Table](/concepts/baseline-table.md) — for `InferenceLog` analysis, this can indicate that a model should be retrained.

^[profile-alerts-databricks-on-aws.md]

## How Alerts Work

1. **Create a query** — Write a Databricks SQL query against the relevant table (e.g., a [Profile Metrics Table](/concepts/profile-metrics-table.md) or [Drift Metrics Table](/concepts/drift-metrics-table.md)). The query should return a value or boolean that the alert will evaluate.
2. **Create the alert** — Use the Databricks SQL alert creation UI or API to attach the alert to the query. You specify the evaluation frequency (how often the query is run) and the trigger condition (e.g., threshold exceeded).
3. **Configure notifications** — By default, email notifications are sent when the alert triggers. You can also integrate with webhooks, Slack, or PagerDuty.
4. **Handle parameters** — If the query uses parameters, the alert uses the default values for those parameters. Ensure the defaults reflect the intended alert condition.

^[profile-alerts-databricks-on-aws.md]

## Relationship to Profile Alerts

Profile alerts are a specialized category of Databricks SQL alerts. They are created and used in exactly the same way as any other alert. The only difference is that the backing query is written against a profile metrics table or drift metrics table, which are generated by [Data Profiling](/concepts/data-profiling.md) in [Unity Catalog](/concepts/unity-catalog.md). ^[profile-alerts-databricks-on-aws.md]
