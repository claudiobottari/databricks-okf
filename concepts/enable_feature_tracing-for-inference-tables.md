# ENABLE_FEATURE_TRACING for Inference Tables

**ENABLE_FEATURE_TRACING** is an environment variable that, when set to `true` on a [Model Serving](/concepts/model-serving.md) endpoint with [Inference Tables](/concepts/inference-tables.md) enabled, instructs the endpoint to log [automatic feature lookup](/concepts/automatic-feature-lookup-for-model-serving.md) DataFrames to that inference table. This allows you to capture the features used during inference for monitoring, debugging, and retraining purposes. ^[configure-access-to-resources-from-model-serving-endpoints-databricks-on-aws.md]

## Requirements

- The serving endpoint must already have inference tables enabled.
- MLflow version 2.14.0 or above must be used on the endpoint.
- The endpoint creator must have the necessary permissions to read from the feature store and write to the inference table. ^[configure-access-to-resources-from-model-serving-endpoints-databricks-on-aws.md]

## Enabling Feature Tracing

You can enable feature tracing when creating or updating a model serving endpoint. The variable can be configured through the Databricks Serving UI, the REST API, or one of the supported SDKs (WorkspaceClient SDK or MLflow Deployments SDK). ^[configure-access-to-resources-from-model-serving-endpoints-databricks-on-aws.md]

### Via the Serving UI

1. In the **Advanced configurations** section of the endpoint creation or update dialog, click **+ Add environment variables**.
2. Enter `ENABLE_FEATURE_TRACING` as the environment variable name.
3. In the value field, enter `true`.
4. Save or update the endpoint. ^[configure-access-to-resources-from-model-serving-endpoints-databricks-on-aws.md]

The feature lookup DataFrame generated by the model’s inference code will then be automatically logged to the endpoint’s inference table.

## Purpose and Benefits

By persisting feature lookups alongside predictions and ground truth, you create a complete record of the features that influenced each model response. This traceability is essential for:

- Auditing model behavior in production.
- Debugging feature drift or data quality issues.
- Reconstructing feature vectors for offline training or retraining.
