# Ray Core to Spark Data Exchange Patterns

**Ray Core to Spark Data Exchange Patterns** describes the three methods for writing data from Ray Core applications to Apache Spark when both frameworks run in the same Databricks environment. These patterns allow teams to leverage Ray Core for custom distributed computations while using Spark for ETL, data management, and integration with [[Delta Lake]] and [[Unity Catalog]]. Databricks supports three exchange strategies to suit different workload sizes, performance requirements, and governance constraints.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

## Pattern 1: Persist output in a temporary location

The most common pattern is to store the output of each Ray remote task in a temporary file (e.g., CSV, Parquet) on DBFS or a [[Unity Catalog Volumes for ML Data|Unity Catalog Volumes|Unity Catalog volume]], then read all files into a single Spark DataFrame. This method works best when the output data is in tabular form, such as a Pandas DataFrame generated by a Ray Core task, and is too large to fit in the memory of the driver node or the shared object store.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

If the output from Ray tasks is too large to fit in driver memory without persisting, consider increasing the memory allocated to the driver node in your Databricks cluster to improve performance.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

```python
import os, uuid, numpy as np, pandas as pd, ray

@ray.remote
def write_example(task_id, path_prefix):
    num_rows = 100
    df = pd.DataFrame({
        'foo': np.random.rand(num_rows),
        'bar': np.random.rand(num_rows)
    })
    df.to_csv(os.path.join(path_prefix, f"result_part_{task_id}.csv"))

n_tasks = 10
dbfs_prefix = "/dbfs/<USERNAME>"
path_prefix = os.path.join(dbfs_prefix, f"/ray_tmp/write_task_{uuid.uuid4()}")
ray.get([write_example.remote(i, path_prefix) for i in range(n_tasks)])

# Read all CSV files into a single Spark DataFrame
df = spark.read.csv(path_prefix.replace("/dbfs", "dbfs:"), header=True, inferSchema=True)
```

## Pattern 2: Connect using Spark Connect

Ray remote tasks can directly connect to the Spark cluster using [[Spark Connect]]. This approach allows a Ray task to write data to a Unity Catalog or Hive table without intermediate files. Each Ray worker creates a remote Spark session via `DatabricksSession`.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

```python
from databricks.connect import [DatabricksSession](/concepts/databrickssession.md)
import ray

@ray.remote
class SparkHandler:
    def __init__(self, access_token=None, cluster_id=None, host_url=None):
        self.spark = ([DatabricksSession](/concepts/databrickssession.md)
                      .builder
                      .remote(host=host_url,
                              token=access_token,
                              cluster_id=cluster_id)
                      .getOrCreate())

    def test(self):
        df = self.spark.sql("select * from samples.nyctaxi.trips")
        df.write.format("delta").mode("overwrite").saveAsTable("catalog.schema.taxi_trips")
        return df.count()

access_token = dbutils.notebook.entry_point.getDbutils().notebook().getContext().apiToken().get()
cluster_id = dbutils.notebook.entry_point.getDbutils().notebook().getContext().clusterId().get()
host_url = f"https://{dbutils.notebook.entry_point.getDbutils().notebook().getContext().tags().get('browserHostName').get()}"

sh = SparkHandler.remote(access_token=access_token,
                         cluster_id=cluster_id,
                         host_url=host_url)
print(ray.get(sh.test.remote()))
```

Because this process funnels all tasks through a single Spark driver, a threading lock is created. Concurrent tasks will execute sequentially. This pattern is best when the number of concurrent Ray tasks is small. For high concurrency, prefer Pattern 1 (temporary location) to avoid contention.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

To use Spark Connect, you must configure the Ray cluster resources to allocate space for Spark. For example, if a worker node has 8 CPUs, set `num_cpus_worker_node` to 7, leaving 1 CPU for Spark. For larger Spark tasks, allocate a larger share of resources.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

For production use, Databricks recommends storing the access token in [[Databricks Secret Scopes|Databricks secrets]] rather than using the notebook-generated token shown in the example.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

## Pattern 3: Third-party libraries

Libraries such as `deltalake` (from the `delta-rs` project) and `deltaray` (from the Delta Incubator) allow Ray Core tasks to write directly to Delta Lake tables from within the task. This approach avoids intermediate file management but currently works only with [[Built-in Hive Metastore|Hive metastore]] tables, not [[Unity Catalog]] tables. Databricks does not officially support these third-party libraries — they are provided as community options.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

Example using `deltalake`:^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

```python
from deltalake import write_deltalake
import pandas as pd, numpy as np, ray

@ray.remote
def write_test(table_name):
    random_df_id_vals = [int(np.random.randint(1000)), int(np.random.randint(1000))]
    pdf = pd.DataFrame({"id": random_df_id_vals, "value": ["foo", "bar"]})
    write_deltalake(table_name, pdf, mode="append")

def main():
    table_name = "database.mytable"
    ray.get([write_test.remote(table_name) for _ in range(100)])
```

Example using `deltaray`:^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

```python
import pathlib, deltaray, deltalake as dl, pandas as pd

cwd = pathlib.Path().resolve()
table_uri = f'{cwd}/tmp/delta-table'
df = pd.DataFrame({'id': [0, 1, 2, 3, 4]})
dl.write_deltalake(table_uri, df)

ds = deltaray.read_delta(table_uri)
ds.show()
```

## Best Practices

- Use the techniques described in the [[Ray Cluster Scaling Best Practices|Ray cluster best practices]] guide to ensure the cluster is fully utilized.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]
- Consider using Unity Catalog Volumes to store output data in a non-tabular format and provide governance.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]
- Set `num_cpus_worker_node` to match the number of CPU cores in the Spark worker node, and `num_gpus_worker_node` to the number of GPUs per Spark worker node. This ensures each Spark worker node launches one Ray worker node that fully utilizes the underlying resources.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]

## Limitations

Unity Catalog currently does not share credentials for writing to tables from non-Spark writers. Therefore, all data written to a Unity Catalog table from a Ray Core task must either be persisted and then read with Spark, or Databricks Connect must be set up within the Ray task.^[combine-ray-and-spark-in-the-same-environment-on-databricks-databricks-on-aws.md]
