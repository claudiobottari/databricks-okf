# Feedback in MLflow Evaluation

**Feedback in MLflow Evaluation** refers to the quality assessments that scorers produce for each input in an evaluation dataset when running `mlflow.genai.evaluate()`. These assessments are attached to individual [Traces](/concepts/traces.md) and can be aggregated into metrics, enabling developers to judge the performance of their GenAI agent configurations. ^[evaluation-runs-in-mlflow-databricks-on-aws.md]

## How Feedback is Collected

Feedback is generated by **scorers** (also called judges) that are passed to the `scorers` parameter of `mlflow.genai.evaluate()`. Each scorer evaluates the agent’s output (and optionally the full execution trace) for a specific quality criterion. The scorers return structured feedback values that are stored as part of the evaluation run. ^[evaluation-runs-in-mlflow-databricks-on-aws.md, create-a-custom-judge-using-make_judge-databricks-on-aws.md]

Example usage: ^[evaluation-runs-in-mlflow-databricks-on-aws.md]

```python
import mlflow

results = mlflow.genai.evaluate(
    data=test_dataset,
    predict_fn=my_app,
    scorers=[correctness_scorer, safety_scorer],
    experiment_name="my_app_evaluations",
)
```

## Feedback Structure

Within an evaluation run, feedback is organized per trace. Each trace corresponds to a row in the evaluation dataset and contains the agent’s inputs, outputs, and a list of feedback objects from all scorers. ^[evaluation-runs-in-mlflow-databricks-on-aws.md]

The evaluation run hierarchy for feedback: ^[evaluation-runs-in-mlflow-databricks-on-aws.md]

```
Evaluation Run
├── Traces (one per dataset row)
│   ├── Trace 1
│   │   ├── inputs: {"question": "What is MLflow?"}
│   │   ├── outputs: {"response": "MLflow is..."}
│   │   └── feedbacks: [correctness: 0.8, relevance: 1.0]
│   ├── Trace 2
│   └── ...
├── Aggregate Metrics
│   ├── correctness_mean: 0.85
│   ├── relevance_mean: 0.92
│   └── safety_pass_rate: 1.0
└── Parameters
    ├── model_version: "v2.1"
    ├── dataset_name: "qa_test_v1"
    └── scorers: ["correctness", "relevance", "safety"]
```

## Feedback Values from Custom Judges

When using [Custom Judges](/concepts/custom-judges.md) created with `make_judge()`, each judge defines its own feedback schema via the `feedback_value_type` parameter. Common feedback value types include `bool`, `int`, `float`, or a predefined set of string labels. ^[create-a-custom-judge-using-make_judge-databricks-on-aws.md]

Example feedback value types from custom judges: ^[create-a-custom-judge-using-make_judge-databricks-on-aws.md]

| Judge | Feedback Values | Purpose |
|-------|-----------------|---------|
| `issue_resolution` | `fully_resolved`, `partially_resolved`, `needs_follow_up` | Assesses outcome quality |
| `expected_behaviors` | `meets_expectations`, `partially_meets`, `does_not_meet` | Checks specific behaviors |
| `tool_call_correctness` | `true`, `false` | Validates tool usage |

These feedback values are recorded per trace and can be compared across different [A/B Comparison of Agent Configurations](/concepts/ab-comparison-of-agent-configurations.md) to determine which variant performs better.

## Aggregate Metrics from Feedback

After all traces are scored, MLflow computes aggregate metrics from the individual feedback values. For example, if a scorer returns a boolean or numeric value, the mean or pass rate may be calculated and stored as an aggregate metric in the evaluation run. ^[evaluation-runs-in-mlflow-databricks-on-aws.md]
