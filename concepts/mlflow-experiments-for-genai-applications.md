# MLflow experiments for GenAI applications

An **MLflow experiment** is the primary organizational container for [GenAI](/concepts/mlflow-genai-evaluate-api.md) applications when using [MLflow Tracing](/concepts/mlflow-tracing.md) on Databricks. Each experiment groups together traces, runs, and metadata for a specific application, providing a centralized view for debugging, monitoring, and comparing performance across iterations. ^[get-started-mlflow-tracing-for-genai-databricks-notebook-databricks-on-aws.md]

## Overview

When developing a GenAI application — such as a [Retrieval Augmented Generation (RAG)](/concepts/retrieval-augmented-generation-rag.md) system, multi-step agent, or conversational assistant — the experiment serves as the logical namespace that collects all tracing data generated by the application's invocations. ^[get-started-mlflow-tracing-for-genai-databricks-notebook-databricks-on-aws.md, get-started-mlflow-tracing-for-genai-in-a-local-ide-databricks-on-aws.md]

## Creating an experiment

Experiments can be created through the Databricks workspace UI. From the **Experiments** page under **AI/ML**, select **GenAI apps & agents** to create a new experiment specifically designed for GenAI workloads. The experiment's ID and path are available from the information icon on the experiment page. ^[get-started-mlflow-tracing-for-genai-in-a-local-ide-databricks-on-aws.md]

## Connecting environments to an experiment

### Databricks notebooks

A Databricks notebook has a default MLflow experiment linked automatically. To associate traces with a specific experiment, call `mlflow.set_experiment()` with the experiment path: ^[get-started-mlflow-tracing-for-genai-databricks-notebook-databricks-on-aws.md]

```python
mlflow.set_experiment("/Shared/tracing-quickstart")
```

### Local IDEs

For local development environments such as VS Code, PyCharm, or Jupyter, the connection requires setting environment variables. From the MLflow experiment UI, select **Log traces locally** to generate the necessary configuration: ^[get-started-mlflow-tracing-for-genai-in-a-local-ide-databricks-on-aws.md]

```bash
export DATABRICKS_TOKEN=<databricks-personal-access-token>
export DATABRICKS_HOST=https://<workspace-name>.cloud.databricks.com
export MLFLOW_TRACKING_URI=databricks
export MLFLOW_REGISTRY_URI=databricks-uce
export MLFLOW_EXPERIMENT_ID=<experiment-id>
```

Then set the experiment in code: ^[get-started-mlflow-tracing-for-genai-in-a-local-ide-databricks-on-aws.md]

```python
mlflow.set_tracking_uri("databricks")
mlflow.set_experiment("/Shared/docs-demo")
```

## Viewing traces

After running an instrumented application, traces appear in the experiment's **Traces** tab in the MLflow UI. Each trace contains a hierarchy of spans — the root span representing the application entry point and child spans representing individual operations such as LLM calls. The UI surfaces metadata including model name, token counts, timing information, inputs, and outputs. ^[get-started-mlflow-tracing-for-genai-databricks-notebook-databricks-on-aws.md, get-started-mlflow-tracing-for-genai-in-a-local-ide-databricks-on-aws.md]

## Benefits for GenAI applications

Experiments provide observability into GenAI application behavior by capturing what was asked, what response was generated, how long each request took, and how many tokens were consumed. For complex multi-step applications, the experiment's trace view reveals the inner workings of each component, making it easier to debug and optimize performance. ^[get-started-mlflow-tracing-for-genai-databricks-notebook-databricks-on-aws.md, get-started-mlflow-tracing-for-genai-in-a-local-ide-databricks-on-aws.md]
