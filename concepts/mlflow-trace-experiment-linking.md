# MLflow Trace Experiment Linking

**MLflow Trace Experiment Linking** refers to the process of associating an [MLflow](/concepts/mlflow.md) experiment with a [Unity Catalog](/concepts/unity-catalog.md) location so that OpenTelemetry trace spans (generated by frameworks such as Langfuse) are automatically stored in a Unity Catalog table under a specified [Catalog and Schema](/concepts/catalog-and-schema.md). This linking is a prerequisite for centralizing trace data on Databricks and enabling governance, query, and analysis capabilities across all traces. ^[export-langfuse-traces-to-databricks-mlflow-databricks-on-aws.md]

## Overview

When traces are produced by an instrumented application (e.g., via the Langfuse `@observe()` decorator), they are sent as OpenTelemetry spans to an OTLP endpoint. For those spans to be persisted in a Unity Catalog table, the MLflow experiment that manages the traces must first be bound to a Unity Catalog schema. This binding is accomplished using `set_experiment_trace_location()`. ^[export-langfuse-traces-to-databricks-mlflow-databricks-on-aws.md]

Once linked, the OTLP exporter uses the table name generated by `set_experiment_trace_location()` in the `X-Databricks-UC-Table-Name` header to route incoming spans to the correct Unity Catalog location. ^[export-langfuse-traces-to-databricks-mlflow-databricks-on-aws.md]

## Benefits

Linking MLflow experiments to Unity Catalog for trace storage provides:

- **Unified access** – Query and compare traces from Langfuse, custom OpenTelemetry, and other frameworks in a single place. ^[export-langfuse-traces-to-databricks-mlflow-databricks-on-aws.md]
- **Scalable analysis** – Use Databricks SQL to analyze trace data at scale. ^[export-langfuse-traces-to-databricks-mlflow-databricks-on-aws.md]
- **Governance** – Apply Unity Catalog access controls, lineage, and data management to all traces. ^[export-langfuse-traces-to-databricks-mlflow-databricks-on-aws.md]

## Usage

### Prerequisites

- A Databricks workspace with the “OpenTelemetry on Databricks” preview enabled.
- An existing MLflow experiment.

### Linking an experiment

Define the Unity Catalog [Catalog and Schema](/concepts/catalog-and-schema.md), then call `set_experiment_trace_location()` to bind the experiment to that location. The function returns a `trace_location` object whose `full_otel_spans_table_name` attribute provides the fully qualified table name used by the OTLP exporter. ^[export-langfuse-traces-to-databricks-mlflow-databricks-on-aws.md]

```python
import mlflow
from mlflow.entities import UCSchemaLocation
from mlflow.tracing.enablement import set_experiment_trace_location

CATALOG_NAME = "<UC_CATALOG_NAME>"
SCHEMA_NAME = "<UC_SCHEMA_NAME>"
EXPERIMENT_ID = "<EXPERIMENT_ID>"

trace_location = set_experiment_trace_location(
    location=UCSchemaLocation(catalog_name=CATALOG_NAME, schema_name=SCHEMA_NAME),
    experiment_id=EXPERIMENT_ID,
)
```

The exporter then includes this table name in the HTTP header when forwarding spans:

```python
# Inside the OTLP exporter configuration:
headers={
    "X-Databricks-UC-Table-Name": trace_location.full_otel_spans_table_name,
}
```

After linking, all trace spans sent from the experiment’s context (e.g., from a Langfuse-instrumented function) are stored in the specified Unity Catalog table and become visible in the MLflow UI under the **Traces** tab. ^[export-langfuse-traces-to-databricks-mlflow-databricks-on-aws.md]
