# Code-based scorers in MLflow GenAI

**Code-based scorers** are custom evaluation functions in [MLflow GenAI](/concepts/mlflow-3-for-genai.md) that allow you to define flexible, programmatic metrics for assessing AI agent or application outputs. Unlike [LLM-as-a-Judge](/concepts/llm-as-a-judge.md) scorers that rely on another language model for evaluation, code-based scorers use Python logic to compute metrics directly from trace data, inputs, outputs, and expectations. ^[code-based-scorer-examples-databricks-on-aws.md]

## Overview

Code-based scorers enable fine-grained, deterministic evaluation of GenAI applications. They can access the full [[MLflow Trace]] object — including spans, inputs, outputs, attributes, and timing information — for detailed metric calculation. Scorers can return primitive values (bool, int, float, str), single [Feedback](/concepts/feedback-object.md) objects, or lists of Feedback objects for multi-faceted assessment. ^[code-based-scorer-examples-databricks-on-aws.md]

## Defining a Code-based Scorer

### Using the `@scorer` Decorator

The primary way to define a code-based scorer is with the `@scorer` decorator from `mlflow.genai.scorers`. The decorated function receives parameters based on the data available and must return a value or [Feedback](/concepts/feedback-object.md) object. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
from mlflow.genai.scorers import scorer
from mlflow.entities import Feedback

@scorer
def my_custom_scorer(outputs: str) -> Feedback:
    # Evaluation logic here
    return Feedback(value=True, rationale="Response meets criteria")
```

### Class-based Scorers (Offline Only)

For scorers that require state or configuration, use the `Scorer` base class. Class-based scorers are Pydantic objects that support additional fields and are instantiated with custom parameters. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
from mlflow.genai.scorers import Scorer
from mlflow.entities import Feedback

class ResponseQualityScorer(Scorer):
    name: str = "response_quality"
    min_length: int = 50
    required_sections: list[str] = None

    def __call__(self, outputs: str) -> Feedback:
        # Evaluation logic
        return Feedback(value=True, rationale="Good quality")
```

**Important:** Class-based scorers are supported only for offline evaluation with `mlflow.genai.evaluate()`. They cannot be registered for [Production Monitoring](/concepts/production-monitoring.md). For production use, apply the `@scorer` decorator instead. ^[code-based-scorer-examples-databricks-on-aws.md]

## Scorer Input Parameters

Code-based scorers can accept any combination of the following parameters, which MLflow automatically injects based on the function signature: ^[code-based-scorer-examples-databricks-on-aws.md]

| Parameter | Description |
|-----------|-------------|
| `trace` | Full [[MLflow Trace]] object with spans, timing, and metadata |
| `inputs` | Input data passed to the GenAI app (e.g., messages dictionary) |
| `outputs` | Response generated by the GenAI app |
| `expectations` | Ground truth values or labels from the evaluation dataset |

## Common Patterns

### Accessing Trace Data

Scorers can inspect trace spans to compute timing-based metrics, such as checking whether LLM response time falls within acceptable limits. ^[code-based-scorer-examples-databricks-on-aws.md]

```python
@scorer
def llm_response_time_good(trace: Trace) -> Feedback:
    llm_span = trace.search_spans(span_type=SpanType.CHAT_MODEL)[0]
    response_time = (llm_span.end_time_ns - llm_span.start_time_ns) / 1e9
    # Compare against threshold and return Feedback
```

### Wrapping Built-in Judges

Custom scorers can wrap predefined [LLM Judges](/concepts/llm-judges.md) like `is_context_relevant` to preprocess data or post-process results. This is useful when the app's input format differs from what the built-in judge expects. ^[code-based-scorer-examples-databricks-on-aws.md]

### Using Expectations

Expectations provide ground truth values for offline evaluation. Scorers can access expectations to perform exact match checks, keyword presence verification, or more complex comparisons against expected outputs. ^[code-based-scorer-examples-databricks-on-aws.md]

### Returning Multiple Feedback Objects

A single scorer can return a list of [Feedback](/concepts/feedback-object.md) objects, each with a unique `name` that becomes a separate metric column in evaluation results. This allows one scorer to assess multiple quality facets simultaneously. ^[code-based-scorer-examples-databricks-on-aws.md]

### Custom LLM Integration

Scorers can call external LLMs for evaluation, giving full control over the judging process. Set the `source` field in the Feedback object to `AssessmentSourceType.LLM_JUDGE` to indicate the assessment source. ^[code-based-scorer-examples-databricks-on-aws.md]

## Error Handling

MLflow surfaces scorer errors gracefully. Scorers can use two approaches: ^[code-based-scorer-examples-databricks-on-aws.md]

1. **Return `AssessmentError`** in a Feedback object for expected failure conditions
2. **Raise exceptions** to let MLflow handle errors automatically

Evaluation continues even when individual scorers fail, preventing one failed scorer from blocking the entire evaluation run. ^[code-based-scorer-examples-databricks-on-aws.md]

## Naming Conventions

Metric names follow these rules: ^[code-based-scorer-examples-databricks-on-aws.md]

- **Primitive return or unnamed Feedback:** The scorer function name becomes the metric name
- **Single named Feedback:** The name specified in the Feedback object is used
- **Multiple Feedback objects:** Each must have a unique name, which becomes the metric name
- **Class-based scorers:** The `name` field of the class determines the metric name

## Chaining Evaluations

Scorers can be chained by using `mlflow.search_traces()` to identify problematic subsets and re-evaluate them with different scorers or an updated app. This iterative approach helps focus on challenging inputs. ^[code-based-scorer-examples-databricks-on-aws.md]

## Conditional Logic

Scorers can apply different evaluation criteria based on input attributes, such as user tier or request type. This enables context-aware assessment using [Guidelines judges](/concepts/guidelines-llm-judges.md) wrapped in custom code. ^[code-based-scorer-examples-databricks-on-aws.md]

## Production Monitoring Considerations

When designing scorers for both offline and [Production Monitoring](/concepts/production-monitoring.md), handle expectations gracefully since production traffic lacks ground truth. Use the `@scorer` decorator pattern for production-compatible scorers. ^[code-based-scorer-examples-databricks-on-aws.md]
