## Feedback Analysis via MLflow SDK

**Feedback Analysis via MLflow SDK** refers to the process of programmatically retrieving, inspecting, and deriving insights from user feedback that has been logged as assessments on traces in [MLflow](/concepts/mlflow.md). The `mlflow-tracing` package provides the necessary APIs to collect user feedback and then analyze it to understand application quality, user satisfaction, and patterns for improvement.

### Prerequisites

- **MLflow 3 is required** – MLflow 2.x is not supported because of performance limitations and missing features essential for production use. ^[collect-user-feedback-databricks-on-aws.md]
- For production deployments, install the `mlflow-tracing` package. In development environments, the full `mlflow` package can be used instead. ^[collect-user-feedback-databricks-on-aws.md]

### Why collect and analyze user feedback

User feedback provides ground truth about an application’s performance. Feedback analysis enables:

- **Real-world quality signals** – Understand how actual users perceive outputs.
- **Continuous improvement** – Identify patterns in negative feedback to guide development.
- **Training data creation** – Build high-quality evaluation datasets from production feedback.
- **Quality monitoring** – Track satisfaction metrics over time and across user segments.
- **Model fine-tuning** – Leverage feedback data to improve underlying models. ^[collect-user-feedback-databricks-on-aws.md]

### Feedback data model

In MLflow, user feedback is captured using the **Feedback** entity, a type of [Assessment](/concepts/assessments.md) that is attached to traces or spans. The Feedback entity stores:

- **Value**: The actual feedback (boolean, numeric, text, or structured data).
- **Source**: Information about who or what provided the feedback – `"HUMAN"` for a user, `"LLM_JUDGE"` for automated evaluation, or `"CODE"`.
- **Rationale**: An optional explanation for the feedback.
- **Metadata**: Additional context such as timestamps or custom attributes.

This structured model integrates seamlessly with MLflow’s evaluation and monitoring capabilities. ^[collect-user-feedback-databricks-on-aws.md]

### Implementing feedback collection

Before analysis, feedback must be collected. Two primary approaches exist:

1. **Using MLflow trace IDs** – The simplest approach: retrieve the trace ID automatically generated by MLflow during request processing and return it to the client. The client later submits feedback referencing that ID via `mlflow.log_feedback()`.
2. **Using client request IDs** – Generate your own unique IDs per request and log feedback later by looking up the trace using that custom ID.

Both approaches store feedback as assessments permanently attached to the trace, visible in the MLflow UI and queryable via the SDK. ^[collect-user-feedback-databricks-on-aws.md]

For streaming responses (e.g., Server-Sent Events), the trace ID is only available after the stream completes. The recommended implementation sends the trace ID as a final event in the stream, and the UI enables feedback controls only after receiving that ID. ^[collect-user-feedback-databricks-on-aws.md]

### Analyzing feedback with the SDK

Once feedback is collected, the `MlflowClient` API provides methods to search traces and inspect their assessments.

#### Retrieving traces within a time window

Use `client.search_traces()` with a filter on `trace.timestamp_ms` to get traces from a specific period. The example below fetches traces from the last 24 hours:

```python
from mlflow.client import MlflowClient
from datetime import datetime, timedelta

def get_recent_traces(experiment_name: str, hours: int = 24):
    client = MlflowClient()
    cutoff_time = datetime.now() - timedelta(hours=hours)
    cutoff_timestamp_ms = int(cutoff_time.timestamp() * 1000)
    traces = client.search_traces(
        experiment_names=[experiment_name],
        filter_string=f"trace.timestamp_ms > {cutoff_timestamp_ms}"
    )
    return traces
```

#### Analyzing feedback patterns

After retrieving traces, use `client.get_trace()` to get full trace details including assessments. The code below computes feedback rate and positive/negative ratio:

```python
def analyze_user_feedback(traces):
    client = MlflowClient()
    total_traces = len(traces)
    traces_with_feedback = 0
    positive_count = 0
    negative_count = 0
    for trace in traces:
        trace_detail = client.get_trace(trace.info.trace_id)
        if trace_detail.data.assessments:
            traces_with_feedback += 1
            for assessment in trace_detail.data.assessments:
                if assessment.name == "user_feedback":
                    if assessment.value:
                        positive_count += 1
                    else:
                        negative_count += 1
    feedback_rate = (traces_with_feedback / total_traces) * 100 if total_traces else 0
    positive_rate = (positive_count / traces_with_feedback) * 100 if traces_with_feedback else 0
    return {
        "total_traces": total_traces,
        "traces_with_feedback": traces_with_feedback,
        "feedback_rate": feedback_rate,
        "positive_rate": positive_rate,
        "positive_count": positive_count,
        "negative_count": negative_count,
    }
```

#### Analyzing multi-dimensional feedback

When feedback includes separate ratings per dimension (e.g., accuracy, helpfulness, relevance), each rating is logged as a separate assessment. The SDK can aggregate scores across dimensions:

```python
def analyze_ratings(traces):
    client = MlflowClient()
    ratings_by_dimension = {}
    for trace in traces:
        trace_detail = client.get_trace(trace.info.trace_id)
        if trace_detail.data.assessments:
            for assessment in trace_detail.data.assessments:
                if assessment.name.startswith("user_") and assessment.name != "user_feedback":
                    dimension = assessment.name.replace("user_", "")
                    ratings_by_dimension.setdefault(dimension, []).append(assessment.value)
    average_ratings = {
        dim: sum(scores) / len(scores)
        for dim, scores in ratings_by_dimension.items() if scores
    }
    return average_ratings
```
